What you're looking at

Three services — Node, a JVM, and CPython — none of which this app has ever seen the source of. Each one is a machine: an independently deployed process that publishes a typed manifest, exactly like a Module Federation remote publishes mf-manifest.json. This app consumes them as plain imported functions — no SDKs, no client libraries, no codegen per language. The federation runtime is the multiplexer; HTTP is just the transport. Watch the cards below flip from cold to warm: nothing attaches until the first call needs it.

// the entire consumer API — types pulled from the machines themselves import { strings } from './machines/java_machine'; import { stats } from './machines/python_machine'; await strings.sha256(text); // runs on the JVM await stats.mean(numbers); // runs on CPython

Anatomy of a call

Import. Bindgen pulled typed bindings from each machine's manifest at dev time — the machine analog of MF's @mf-types. The import costs nothing.
First call → attach. The runtime plugin claims the machinen+http:// entry, fetches the manifest, negotiates the semver version — the card flips to warm.
Every call → RPC. Arguments cross as JSON, governed by deadlines, retries, and a circuit breaker. The guest runs your function and answers.
Hooks see everything. The activity log below is the plugin's hook system narrating boots, calls, latencies, and failures in real time.

Polyglot pipeline — one request, three machines
