This page never touches federation — the browser only calls fetch('/api/…'). Federation lives in the host: a Node process (apps/host/src/server.ts) that is a stock Module Federation runtime consumer — the same createInstance + loadRemote as module-federation.io. The twist: every remote entry is a machinen address, claimed by a runtime plugin that resolves it to a running machine — a Node service, a JVM, a CPython process — and hands back typed async functions. Every card below answers one question — what moves where? — and shows the exact bytes that crossed each boundary: open “what just happened on the wire” after you click.

the host, verbatim — apps/host/src/server.ts (runs in the host process, never the browser)
import { createInstance } from '@module-federation/runtime'; import { machinenPlugin, httpAttachDriver } from '@federated-compute/machinen-plugin'; // three entry kinds, one runtime: // machinen://<path> boot from local disk // machinen+http://host:port attach to a deployment // machinen+pull+http://… fetch the artifact, boot a local clone const host = createInstance({ name: 'host', remotes: [{ name: 'java_machine', entry: 'machinen+http://127.0.0.1:3802' }, /* … */], plugins: [machinenPlugin({ driver: httpAttachDriver() })], }); // loadRemote, exactly like any federated module — but it's a JVM const strings = await host.loadRemote('java_machine/strings'); await strings.sha256(text);

The four ways to move compute — what moves where?

attach hostmachine Calls move. The machine stays where it was deployed; every invocation is an RPC to its address. acts 02 · 03 · 05 below
freeze / restore state.snap Nothing crosses the network. State drops to disk, the process dies, a new one resumes mid-count. act 04 · steps 1–3
pull to me artifacthost The artifact moves to you. Image or warm snapshot is fetched over HTTP; an independent clone boots locally. act 04 · steps 4–5
code → data artifactdata region The artifact moves toward the data. Deploy by pull into the far region; calls stay short forever after. page 02 — data gravity

act 04 · Machine lifecycle — freeze it, restore it, fork it by pull

freeze / restore state.snap steps 1–3 — nothing moves, state survives death pull to me artifacthost steps 4–5 — the artifact moves, a clone boots

One machine, one stepper, the whole lifecycle. Steps 1–3: snap_machine is booted from an image (process driver, fixed port 3811), worked to 3, frozen into a .snap bundle, killed, and restored by pointing the same remote name at the bundle. Steps 4–5: the restored process serves /mf-manifest.json, /mf-image and /mf-snapshot — it is a registry of itself — so a machinen+pull+http:// entry forks it: snapshot re-fetched fresh each time (it's no-store), image by digest — MISS then HIT. Clones own their state; divergence is the point. Attach to state you share; pull state you want to own.

runs in the host the fork is just another remote entry
await plugin.snapshotMachine('snap_machine'); // 2 · freeze -> .snap host.registerRemotes([{ name: 'snap_machine', entry: 'machinen://….snap?port=3811' }], { force: true }); // 3 · restore host.registerRemotes([{ name: 'snap_clone_a', // 4 · fork by pull entry: 'machinen+pull+http://127.0.0.1:3811?artifact=snapshot' }]); const counter = await host.loadRemote('snap_clone_a/counter'); // pulls, verifies, boots the clone
