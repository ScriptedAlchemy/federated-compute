This page never touches federation — the browser only calls fetch('/api/…'). Federation lives in the host: a Node process (apps/host/src/server.ts) that is a stock Module Federation runtime consumer — the same createInstance + loadRemote as module-federation.io. The twist: every remote entry is a machinen address, claimed by a runtime plugin that resolves it to a running machine — a Node service, a JVM, a CPython process — and hands back typed async functions. Every card below answers one question — what moves where? — and shows the exact bytes that crossed each boundary: open “what just happened on the wire” after you click.

the host, verbatim — apps/host/src/server.ts (runs in the host process, never the browser)
import { createInstance } from '@module-federation/runtime'; import { machinenPlugin, httpAttachDriver } from '@federated-compute/machinen-plugin'; // three entry kinds, one runtime — code only moves between machines over HTTP: // machinen+http://host:port attach to a deployment (calls move, code doesn't) // machinen+pull+http://… fetch the artifact over HTTP, boot a local clone // machinen://<path> boot from a local path (deployments booting their own code) const host = createInstance({ name: 'host', remotes: [{ name: 'java_machine', entry: 'machinen+http://127.0.0.1:3802' }, /* … */], plugins: [machinenPlugin({ driver: httpAttachDriver() })], }); // loadRemote, exactly like any federated module — but it's a JVM const strings = await host.loadRemote('java_machine/strings'); await strings.sha256(text);

act 04 · Machine lifecycle — freeze it, restore it, fork it by pull

freeze / restore state.snap steps 2–3 — nothing moves, state survives death pull to me artifacthost steps 1, 4–5 — the artifact moves, a clone boots

One machine, one stepper, the whole lifecycle — and the host never reads machine code from disk. Step 1: snap_machine's image is pulled over HTTP from compute_machine's published /mf-image (digest MISS — the code crosses the wire exactly once) and booted from the host's own cache (process driver, fixed port 3811), then worked to 3. Steps 2–3: frozen into a .snap bundle (state captured over HTTP via /mf/state), killed, and restored. Steps 4–5: the restored process serves /mf-manifest.json, /mf-image and /mf-snapshot — it is a registry of itself — so a machinen+pull+http:// entry forks it: snapshot re-fetched fresh each time (it's no-store), image already cached by digest — HIT, near-instant. Step 5 also pins ?digest= to the exact image step 4 pulled. Clones own their state; divergence is the point. Pulls are deployment traffic — resolver-level fetches that bypass the per-machine call circuit breaker by design. Attach to state you share; pull state you want to own. One honest limit: this lane only moves what the guest chose to put in dehydrate() — the hot solver in the whole-VM lane below is deliberately not in it, so freeze/restore here would lose that heap.

whole VM — machinen driver

checking hardware…

Same federation, zero cooperation. A hot route solver runs inside a real microVM; its heap — memo cache, RNG state, iteration count, a ticking setInterval — is not in dehydrate(). The app-state lane above cannot move it. This lane moves the whole heap; nothing opted in: plugin.publishMachine() dumps the VM mid-flight into a content-addressed machinen-vmstate@1 bundle (GB-scale, arch-bound), and a second MF instance pulls it with a machinen+pull+…?artifact=vmstate entry and resumes it — heap, timers, kernel — without the guest ever being asked.

act 06 · Activity — runtime plugin hooks, live

Observable by construction. Every boot, call, snapshot, artifact pull, crash, and circuit-breaker event flows through the plugin's hook system — this log is just a hook listener in the host. The failure events are not theoretical: cause one yourself.

machine alive
The kill is real: one federated call to an exposed admin function makes the machine process.exit(), then a burst of calls hits the dead process — watch onMachineCrash flip the card cold and the circuit breaker open. The host cannot restart someone else's deployment (containment); the demo orchestrator supervises the process and brings it back, and the first successful call re-attaches.
