The setup

The database machine lives in eu-west. The consumer (this app) runs in us-east. A "top spenders" report needs 1 + N queries (list users, then orders per user). Same report, two topologies: query the database across the WAN per query — or make one federated call to an analytics machine deployed next to the database, which runs the same queries over same-region hops. The consumer's code style is identical either way: imported functions. Only the federation entries (addresses) differ.

The lesson

Latency is a property of topology, not of code. Both scenarios run byte-identical business logic — the same 1 + N queries, the same answer. The only difference is where the loop runs relative to the data. Chatty code next to far data is the most common distributed-systems performance bug there is (the N+1 problem, at region scale).

Federation makes topology a config change. The consumer imports analytics.topSpenders() the same way it imports db.ordersFor() — typed functions pulled from machine manifests. Moving the computation from "here" to "next to the data" changed an address in an env var, not a line of consumer code. The analytics machine itself is a consumer too: it reaches db_machine through the very same bindings, just resolved to a local address. Machines federating machines.

With Machinen, "moving the code" is literal. Today this demo deploys the analytics service in the data region. With Machinen, you'd snapshot the running analytics machine — heap, state, open connections — and restore it next to the database (or fork it into every region you have). Boot once, run where the data lives. The federation entry is the only thing that changes.
