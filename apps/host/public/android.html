The setup — act 08: freeze an entire device

Page 01's lifecycle card froze a process (an app-level .snap bundle). This page runs the same arc against a whole operating system: a real machinen microVM (KVM) boots Debian, installs QEMU, and powers on an Android-x86 4.4 device inside itself — emulated instruction by instruction, no nested KVM. An app is launched over adb. Then the outer VM is frozen into a vmstate bundle: guest RAM, vCPU state, disk — which means the Android kernel, the running app, the adb daemon, and every TCP connection between them are bytes in a directory. Restoring the bundle thaws all of it mid-instruction.

Honesty notes: the machinen guest exposes no nested KVM and a single vCPU, so Android runs under pure TCG emulation — the first power-on builds the device from scratch (~4–12 minutes end to end, ISO download included). The lab then eats its own dogfood: the booted device is cached as a golden vmstate bundle, and every later power-on restores it in ~25 seconds — boot once, restore forever. The outer VM is capped at 1880 MiB because machinen 0.4.0 reads the vmstate back with a single readFileSync (2 GiB buffer ceiling) — and a vmstate is essentially a full RAM image. adbd is enabled from the device's serial root console because Android 4.4 only auto-starts it on USB gadget events, and an emulated PC has none.

The arc — boot · launch · freeze · resume

1

Power on the device

First time: boot the microVM, install qemu+adb, fetch the Android-x86 ISO, power on the emulated device, wait for sys.boot_completed=1, enable adb over TCP — then cache the booted device. Every later power-on restores that golden bundle in seconds.

2

Launch the app

am start the Settings app, then capture the evidence that matters: app pid, kernel boot_id, uptime — and a screenshot straight off the framebuffer.

3

Freeze the whole VM

One vmstate snapshot captures Android mid-frame, then the source VM is killed. The running device exists only as a ~1.9 GiB bundle on disk.

4

Resume

Restore the bundle into a fresh VM. No adb reconnect needed — the adb server, its TCP connection, and adbd all thawed together. Same pid, same boot_id, uptime continued.

The lesson

The freeze doesn't know what it's freezing. Page 01 snapshotted a counter's heap; this page snapshots a kernel, a window manager, a Dalvik VM, and a TCP mesh — with the same one-line operation. vmstate capture works at the layer below software, so the software above it cannot tell the difference. The app's uptime simply keeps counting.

The evidence is the OS's own bookkeeping. Same pidof, same kernel boot_id (a fresh boot would mint a new one), monotonic uptime that continued rather than reset — and two screenshots of the same app instance straddling its own nonexistence.
